# NOV Navi 社員属性依存棚卸し 2026-07-02

## 結論

NOV Navi側では、部署・役職・職種・雇用形態・就労ステータスを独自マスタ化していません。

現時点で実装上の依存があるのは、主に以下です。

- 問い合わせルート `concierge_department_routes`
- 問い合わせ履歴 `concierge_department_inquiries`
- HUB Context設計上の `roleKeys` / `storeAssignments`

`job_types`、`employees.job_type_id`、`employment_type`、`employment_status`、`leave_type` を参照して動くNOV Navi実装はありません。

## 1. 問い合わせルートで `departments` に依存している箇所

### 実装上の状態

`public.departments.id` への直接FKや参照はありません。

現在の問い合わせルートは、NOV Navi内の問い合わせ導線として `public.concierge_department_routes` を使っています。

対象ファイル:

- `supabase/concierge_20260701_department_inquiries.sql`
- `supabase/functions/concierge-api/index.ts`
- `app.js`
- `index.html`
- `OS_NotificationEngine連携仕様_NOVNavi.md`

### テーブル

```text
public.concierge_department_routes
  id text primary key
  department_name text not null
  owner text
  is_active boolean
  sort_order integer
```

```text
public.concierge_department_inquiries
  route_id text references public.concierge_department_routes(id)
  store_id uuid references public.stores(id)
  employee_id uuid references public.employees(id)
  phase1_login_id text
  status text
```

### 初期ルート

```text
hr
accounting
education
sales
fc
system
```

### 重要な判断

この `department_name` はCore DBの部署正本ではなく、NOV Navi画面上の問い合わせ先ラベルです。

現時点では `target_type = global`, `purpose = concierge.department_inquiry.{route_key}` でOS Notification Engineへ接続する設計です。

### 未対応 / 注意

将来 `target_type = department`, `target_id = public.departments.id` へ寄せる設計メモがありますが、Core DBの部署整理完了までは実装しません。

`concierge_department_routes.department_name` は名前が紛らわしいため、将来DDLを触るタイミングで `route_label` などへの改名を検討できます。ただし現時点では既存DDL投入済みのため、互換性優先で変更しません。

## 2. HUB Contextで社員属性を参照している箇所

### 実装上の状態

HUB Context対応は設計資料段階です。現行フロント/Edge FunctionはまだPhase1店舗ID/PASSログインが主です。

対象ファイル:

- `HUB_Context対応設計_NOVNavi.md`
- `NOVNavigator_HUB組み込み引き継ぎ.md`
- `README.md`
- `OPERATION_CHECKLIST.md`

### 参照予定の属性

```text
employeeId        -> public.employees.id
firebaseUid       -> public.employees.firebase_uid
roleKeys          -> public.roles / public.employee_roles
storeAssignments  -> public.stores.id ベースの店舗所属
activeStoreId     -> public.stores.id
```

### 参照しない属性

HUB Context設計では、以下をNOV Navi側の判定条件として使っていません。

```text
departments.id
positions.id
job_types.id
employees.employment_type
employees.employment_status
employees.leave_type
```

部署名、役職名、店舗名などの表示補助は受け取ってよいが、正本として保存しない方針です。

### 管理画面権限

管理画面は社員属性ではなく `roleKeys` で制御します。

候補:

```text
super_admin
admin
executive
backoffice
nov_navi.admin
```

`department_manager` は部門別管理の候補として記載がありますが、全体管理者とは分ける前提です。未確定の部署値へ直接固定しません。

## 3. `job_types` 未投入に関する対応

### 現状

NOV Navi実装には `job_types` 参照はありません。

`CoreDB_DDLレビュー依頼_職種マスタ追加候補.md` に職種マスタ追加の設計メモがありますが、これはCore DBレビュー用であり、NOV Navi側DDLではありません。

### 未対応として止めるもの

- NOV Navi側で `job_types` を作らない
- `employees.job_type_id` を前提にしたUI分岐を作らない
- 美容師、レセプション、カラーリストなどの職種別問い合わせルートをNavi側で固定しない
- `employment_type` と職種の混在値をNavi側で補正しない

## 4. 役職・雇用形態・就労ステータス・休職種別

### 実装依存

現行NOV Naviには、以下を参照して分岐する実装はありません。

```text
public.positions
employees.employment_type
employees.employment_status
employees.leave_type
```

### 資料上の記載

READMEやHUB引き継ぎ資料では、社員属性の正本として以下を記載しています。

```text
部署 = public.departments.id
役職 = public.positions.id
職種 = public.job_types.id 新設後
雇用形態 = employees.employment_type
就労ステータス = employees.employment_status
休職種別 = employees.leave_type
権限 = public.roles / public.employee_roles
```

これは「正本はCore DB」という方針メモであり、現時点でNOV Navi側が値を更新・補正・独自管理するものではありません。

## 6. NOV Navi側で守ること

- NOV Navi側で `job_types` を作らない
- 部署、職種、役職マスタをNOV Navi専用に作らない
- 問い合わせルートを未確定の `departments.id` に固定しない
- 既存社員属性をNOV Navi側から一括更新しない
- LINE WORKS通知先正本をNOV Navi側に持たない
- 権限判定は `roles / employee_roles` 由来の `roleKeys` に寄せる

## 7. CORE / OSへの返答

NOV Navi側の棚卸し結果として、社員属性整理未反映による即時ブロッカーはありません。

ただし、問い合わせルートには `department_name` という表示ラベルがあるため、これはCore DB `departments` の正本ではないことを運用・設計上明記します。

Core DB側の `job_types`、`employees.job_type_id`、雇用形態、就労ステータス、休職種別の整理が完了するまでは、NOV Naviでは職種・部署・役職・雇用形態・就労ステータスを使った分岐や自動通知先決定を実装しません。
