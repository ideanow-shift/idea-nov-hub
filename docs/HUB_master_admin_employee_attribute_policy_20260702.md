# HUBマスタ管理 正式指示: 社員属性の分離ルール

## 結論

HUBマスタ管理では、社員属性を以下の軸に分けて管理する。

```text
部署 = departments
役職 = positions
職種 = job_types
雇用形態 = employees.employment_type
就労ステータス = employees.employment_status + employees.leave_type
権限 = roles / employee_roles
```

最重要ルールは、**役職と職種を混ぜない**こと。

- 役職: 店長、エリアマネージャー、部長、社長
- 職種: 美容師、レセプション、カラーリスト、本部スタッフ
- 雇用形態: 正社員、パート・アルバイト、業務委託
- 権限: store_manager、area_manager、accounting など

## 1. 部署

基本候補:

- 取締役会
- 営業部
- 教育部
- EC事業部
- 総務人事部
- 経理部
- カラー専門店事業部

正式追加対象:

- カラー専門店事業部

追加検討候補:

- 情報システム / システム管理
- 経営企画 / 経営管理
- マーケティング / 広報

ただし、無理に増やさず、実際に所属者または責任者がいる部署だけを登録する。

## 2. 役職

役職は、組織上の責任・肩書きだけを扱う。

標準候補:

- 相談役
- 会長
- 社長
- 副社長
- 取締役
- 執行役員
- 部長
- 課長
- 係長
- エリアマネージャー
- 店長
- 店長見習い
- 副店長
- FCオーナー
- FCオーナー見習い
- 一般スタッフ

正式追加対象:

- 相談役
- 課長
- 一般スタッフ

`未設定` はDBの正式な役職にしない。NULLを画面上で「未設定」と表示する。

## 3. 職種

職種は、シフト自動生成・人件費・採用・教育分類で使う。

標準候補:

- 美容師
- レセプション
- カラーリスト
- 本部スタッフ
- その他

`本部` は職種ではなく所属・部署寄りのため、職種として使う場合は `本部スタッフ` にする。

## 4. 雇用形態

標準候補:

- 役員
- 正社員
- パート・アルバイト
- 業務委託
- その他

`レセプションパート` は廃止し、以下のように分離する。

```text
employment_type = パート・アルバイト
job_type = レセプション
```

## 5. 就労ステータス

`産休` と `育休` は就労ステータスではなく、休職種別として扱う。

employment_status:

- 現職
- 休職
- 退職
- 出向

leave_type:

- 産休
- 育休
- 傷病
- 介護
- その他

これにより、休職者数、育休中人数、産休復帰予定、離職率を正しく集計できる。

## 6. 権限

権限は `roles / employee_roles` で管理する。

役職と権限は混ぜない。

例:

```text
position = 店長
role = store_manager
job_type = 美容師 または レセプション
employment_type = 正社員
```

## 7. 禁止ルール

以下のような家族関係・敬称ラベルは、社員属性に入れない。

- 会長夫人
- 創業者夫人
- 夫人
- その他、血縁・配偶者・敬称を表すだけの値

入れてはいけない場所:

- departments
- positions
- job_types
- employees.employment_type
- employees.employment_status
- employees.leave_type
- roles / employee_roles

必要な文脈がある場合は、社員属性ではなく備考・メモ・履歴文脈で扱う。

## 8. 変更履歴

部署・役職・職種・雇用形態・就労ステータス・権限を変更した場合は、`master_change_logs` に履歴を残す。

## 9. HUB側で守ること

- 各アプリごとに社員属性マスタを作らない
- 社員名・店舗名・部署名・役職名をアプリ側で正本保存しない
- HUB master-adminを社員属性管理の入口にする
- 表示名はCore DBから参照する
- 役職と職種を混ぜない
- 職種と雇用形態を混ぜない
- 役職とシステム権限を混ぜない
- `未設定` はDB値にせずNULL表示にする
