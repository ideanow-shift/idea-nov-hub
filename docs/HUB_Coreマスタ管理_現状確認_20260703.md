# HUB Coreマスタ管理 現状確認 2026-07-03

## 目的

このメモは、NOV HUB Core・マスタ管理チャットの初期整理として、HUB master-admin と `nov-hub-api` が現在どのCore情報を扱っているかを確認するものです。

このメモでは調査・整理のみを行い、DDL / UPDATE / DELETE / 一括更新は実行しません。

- 社員名簿

## master-admin通常画面の管理対象

### 社員

画面:

- 社員一覧
- 社員詳細
- HUB基本権限
- ログイン / PIN管理
- アプリ別権限
- プロフィール画像
- Firebase UID連携

主な参照元:

- `public.employees`
- `public.departments`
- `public.positions`
- `public.job_types`
- `public.stores`
- `public.employee_roles`
- `public.employee_login_credentials`
- `public.employee_profile_images`
- `public.master_change_logs`

保存・更新:

- `masterCreateEmployee`
- `masterUpdateEmployee`
- `masterAssignDefaultStaffRole`
- `masterUpdateEmployeeAppRoles`
- `masterUpdateEmployeeLoginCredential`
- `masterUploadEmployeeProfileImage`
- `masterLinkFirebaseUid`

補足:

- メールアドレスはログイン / PIN管理パネル側の1か所で入力する構成。
- `staff` 付与時、未保存の社員情報がある場合は先に社員情報を保存してから付与する。
- `positions` については、家族関係・敬称ラベルを保存不可にするAPIガードあり。
- 正式役職リスト以外の保存ガードは、既存整理後に維持・強化する対象。

### 店舗

画面:

- 店舗一覧
- 店舗詳細
- LINE WORKS通知先

主な参照元:

- `public.stores`
- `public.corporations`
- `public.business_units`
- `os.notification_destinations`
- `public.master_change_logs`

保存・更新:

- `masterUpdateStore`

補足:

- LINE WORKSチャンネルIDは `stores` 直持ちではなく、`os.notification_destinations` を利用。
- `provider = line_works`
- `target_type = store`
- `purpose = expense_approval`

### アプリカード

画面:

- アプリ一覧
- アプリカード編集
- アプリ追加

主な参照元:

- `public.portal_apps`
- `public.master_change_logs`

保存・更新:

- `masterListPortalApps`
- `masterUpdatePortalApp`
- `masterCreatePortalApp`

補足:

- `app_id`, `app_name`, `description`, `url`, `category`, `icon`, `color`, `required_level`, `allowed_tags`, `target_department`, `target_position`, `is_active`, `is_featured`, `priority` を扱う。
- HUBカード表示の正本は `portal_apps` に寄せる方針。

### 権限参照

画面:

- 権限タブ
- 社員詳細内のアプリ別権限

主な参照元:

- `public.roles`
- `public.employee_roles`

補足:

- HUBから各アプリへ渡す `roleKeys` の正本は `employee_roles`。
- 現状は IDEA LINK 権限表示・編集を中心に扱っている。
- 大規模なrole_key追加・削除・命名変更はOSレビュー必須。

### プロフィール画像

画面:

- 社員詳細内のプロフィール画像

主な参照元:

- Supabase Storage private bucket `employee-profile-images`
- `public.employee_profile_images`

保存・更新:

- `masterUploadEmployeeProfileImage`

補足:

- `employee_profile_images` へmetadataを保存。
- 表示は署名URLをbackend側で発行。
- signed URLはDBへ保存しない。
- 最終的にはHUB master-adminをプロフィール画像登録UIの正本にする。

## HUB context / bootstrap のCore情報

現在のHUB contextは、各アプリへ渡す非secretの社員コンテキストとして扱う。

主な安定項目:

- `employeeId`
- `coreEmployeeId`
- `supabaseEmployeeId`
- `staffId`
- `employeeNumber`
- `name`
- `email`
- `authEmail`
- `corporation`
- `department`
- `position`
- `primaryStore`
- `employmentStatus`
- `employmentType`
- `roleKeys`
- `tags`

重要ルール:

- `employeeId`, `coreEmployeeId`, `supabaseEmployeeId`, `staffId` は Core DB `employees.id` を指す。
- `employeeNumber` は人間が見る社員番号。
- HUB contextは表示・導線・権限ヒントであり、セキュリティ境界ではない。
- 書き込み系アプリは、必ずserver-side endpoint / RLS / Edge Function側で再検証する。

今後確認する項目:

- `job_type` / `jobType` / `jobTypeId` / `jobTypeName` をHUB contextに安定項目として含めるか。
- `storeAssignments` を標準項目として明文化するか。
- app別に必要な最小contextを整理するか。
